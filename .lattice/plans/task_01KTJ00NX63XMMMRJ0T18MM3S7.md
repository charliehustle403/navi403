# NAVI-4: Milestone 1 — Skeleton + DB

## Scope (build spec §11 milestone 1)
FastAPI app skeleton + Postgres data layer + Alembic migration + `GET /health` + Docker
Compose Postgres. CLAUDE.md already exists (NAVI-2) — not in scope. NO broker/tools/loop/router
yet (those are M2/M3). Stay strictly read-only and within the MVP.

## Decisions (locked)
- Package manager: **uv**; Python 3.12; **src layout** (`src/navi/`).
- DB: **Docker Compose Postgres 16** for local dev. Driver: **psycopg v3** (sync). DATABASE_URL
  = `postgresql+psycopg://navi:navi@localhost:5432/navi`.
- DB access: **sync** SQLModel sessions; FastAPI endpoints as plain `def` (threadpool). Simplest
  correct option for the MVP; no async footguns.
- App binds to **127.0.0.1** (spec §6.8 MVP rule). uvicorn entry `navi.api:app`.

## Files to create
- `pyproject.toml` — project + deps. runtime: fastapi, uvicorn[standard], sqlmodel, alembic,
  psycopg[binary], pydantic-settings, httpx, anthropic. dev: pytest, ruff, mypy. Tool config for
  ruff + mypy + pytest here.
- `docker-compose.yml` — Postgres 16 service (user/pw/db = navi; volume; port 5432).
- `.env.example` — ANTHROPIC_API_KEY=, DATABASE_URL=postgresql+psycopg://navi:navi@localhost:5432/navi
- `src/navi/__init__.py`
- `src/navi/config.py` — pydantic-settings `Settings` reading `.env` (anthropic_api_key, database_url).
- `src/navi/db.py` — engine from settings; `get_session()` dependency; `init`/metadata helpers.
- `src/navi/models.py` — SQLModel tables for the FULL spec §5 schema:
  agents, tools, **agent_tools** (join), runs, **trace_events** (with tool_name/route/verdict/
  tokens_in/tokens_out/payload_hash), approvals, memory_candidates, memories. Types/enums per §5;
  `may_influence_actions` default false; unused-in-MVP columns present (additive).
- `src/navi/api.py` — FastAPI `app`; `GET /health` -> `{status:"ok", db:"ok"|"error"}` (pings DB
  with `SELECT 1`, never 500s on db-down — reports db:"error"). `GET /` optional ping.
- `alembic.ini` + `migrations/env.py` (reads DATABASE_URL from env; target_metadata = SQLModel
  metadata via importing models) + initial autogenerated revision creating all §5 tables.
- `tests/test_health.py` — TestClient smoke: `/health` returns 200 with `status` key (DB-agnostic
  so it passes in CI without a live DB); a second DB-up assertion guarded/skipped if DB unreachable.
- Update `README.md` only if needed (it already documents scope) — likely leave as-is. Append a
  short "Local dev (Milestone 1)" run section to CLAUDE.md Commands? No — keep CLAUDE.md commands
  as-is (already accurate-enough); put run steps in a new `GETTING_STARTED`? No — out of scope.
  Minimal: a `docker compose up` + `uv run` note can go in README's Status section if clean.

## Verification (must run before commit)
1. `uv sync` resolves and installs.
2. `docker compose up -d` brings Postgres 16 up; wait for healthy.
3. `uv run alembic upgrade head` applies the initial migration; verify all 9 tables exist
   (`agents, tools, agent_tools, runs, trace_events, approvals, memory_candidates, memories`).
4. `uv run uvicorn navi.api:app` (or TestClient) -> `GET /health` returns 200 `{status:ok, db:ok}`.
5. `uv run pytest` green. `uv run ruff check .` clean. `uv run mypy src` clean.
If Docker isn't running, scaffold + commit may proceed but MUST clearly report which verification
steps were skipped (no false "verified").

## Acceptance criteria
- All files above exist; project installs via `uv sync`.
- Alembic initial migration creates the full §5 schema incl. `agent_tools` and fattened
  `trace_events`; `alembic upgrade head` succeeds against Docker Postgres 16.
- `GET /health` returns 200 and reports DB connectivity without 500-ing when DB is down.
- pytest green; ruff + mypy clean.
- Binds to localhost. No broker/tools/loop yet. No secrets committed (.env ignored; only .env.example).
